## 2. Core Problem

**Problem statement.** Traditional online learning is built around video lectures and multiple-choice quizzes. Both treat the learner as a passive recipient. Cognitive science (and any honest teacher) knows the deepest learning happens through **conversation**: explaining ideas in your own words, getting Socratic pushback, defending positions, debugging mistakes, and being asked penetrating follow-up questions. Until LLMs, simulating that one-on-one conversational instruction at scale was impossible. With LLMs it is — but existing LMS platforms (Canvas, Blackboard, Moodle, Teachable, Thinkific, Kajabi) bolt chatbots on as a side feature. They aren't built around them.

**Who has this problem.**
- University instructors who know discussion drives learning but can't give every student one-on-one Socratic time
- Corporate L&D teams running technical, sales, or compliance training who need real practice, not click-through compliance theatre
- Subject-matter experts (consultants, coaches, healthcare educators, legal trainers) who want to package their expertise as repeatable, scalable instruction
- Instructional designers tired of building courses out of slides and form-quizzes
- Students/learners who get more out of conversation than passive video, but rarely have access to it

**Why it matters.** Active learning beats passive learning by every educational metric (retention, transfer, motivation). Chatbots can simulate active learning at infinite scale and zero marginal cost. The bottleneck is no longer the medium; it's the **authoring experience** — designing pedagogically-sound chatbot conversations, sequencing them, grading transcripts, and proving completion. That's the gap.

**What happens if not solved.** Educators stay stuck choosing between (a) high-touch low-scale (in-person discussion, expensive consultants), (b) low-touch high-scale (videos + quizzes, low retention), or (c) the rapidly expanding wild-west of "ChatGPT homework" without any instructional framing, grading, or certification. Institutions either lose ground to AI-native upstarts or end up paying enterprise vendors for shoehorned chatbot features.

**How users currently solve it.** Mostly badly. Combinations of:
- Ad-hoc ChatGPT in the browser with hand-written system prompts that leak to learners
- Custom-built tutor sites (rare, expensive, brittle)
- LLM features bolted onto Canvas / Moodle / Blackboard (limited authoring, no grading workflow, weak rubrics)
- Single-vendor AI-tutor products that don't compose into multi-step learning paths
- Plain LLM chat with no rubric, no transcript grading, no completion proof, no multi-tenant org management

## 4. Product Vision

**Long-term vision.** Become the default platform for AI-tutor-driven instruction — the LMS that's chatbot-native rather than chatbot-bolted-on. Where any instructor, coach, expert, or training department can take their pedagogy and turn it into scalable conversational learning, with verifiable completion. The "Notion for AI tutoring" or the "Canvas for chatbot-native courses."

**What it should eventually become.** A platform with:
- A polished visual builder where designing a Chatrail feels like designing a flowchart
- A library / marketplace of starting templates ("debate coach", "medical history simulator", "constitutional law moot court", "junior-engineer code review") shareable across the community
- LMS interoperability via LTI 1.3 so it slots into Canvas / Blackboard / Moodle without ripping them out
- Native SCORM / xAPI / certificate integrations so corporate buyers don't have to convince their compliance team
- Rich analytics on learning outcomes (mastery curves, common-misconception detection from transcripts, teacher-facing insights from real conversations)
- A serious bring-your-own-LLM-key story for cost-sensitive enterprise buyers
- Multi-language support so the same pedagogy ships globally

**What makes it useful (today's value prop).** It removes the friction between "I have a teaching idea worth scaling" and "my learners are practicing it conversationally with AI." Authoring, grading, rostering, billing, and certification are all in one platform built specifically around the chatbot-as-lesson model.

**What makes it different from a generic AI wrapper.**
- **Pedagogical structure.** Every bot has a rubric, a grading workflow, an attempts policy, and a place in a sequenced learning path. It's not a chat box.
- **Multi-tenant SaaS architecture.** Real orgs, seat licensing, RLS, role-aware UI, audit logs, plan gating.
- **Transcript-as-artifact grading.** Conversations become gradable submissions. AI suggests rubric scores; instructors override.
- **Visual path builder.** Real React Flow graph with conditional unlocks, branching, milestones, and cert nodes.
- **Verifiable certificates.** Per-org template, signature, logo, unique verification code, public verification URL.
- **Provider-agnostic.** Anthropic / OpenAI / Gemini behind one adapter. Per-bot model selection. Future BYO-key path.
- **A confident, opinionated visual identity.** The brutalist B&W design is a deliberate "we are not your enterprise LMS" signal.
- **Cost transparency.** Live per-attempt cost estimate in the bot editor; org-admin sees MTD cost on the dashboard; usage logs ledger every LLM call.
- **One-prompt Chatrail generator** (just shipped) — describe a course in plain English, get a 3–7 node draft with ready-written system prompts.

**Ideal user experience.**
- A trainer types a paragraph describing what they want learners to walk away knowing
- A draft Chatrail materializes with prewritten bots, rubrics, and a flow
- They tweak each bot inline with a live PREVIEW CHAT panel
- They publish, invite learners, and watch progress on a real dashboard
- Learners feel like they're talking to a knowledgeable, slightly-strict tutor, not a generic chatbot
- Grading takes minutes per submission with the AI summary + suggested rubric scores doing the first pass
- Certificates issue automatically and verifiers can confirm them with a public URL

## 9. User Roles and Permissions

The role enum lives at `app.user_role` in `00010000000002_identity.sql`:

```
super_admin | org_admin | instructor | ta | learner
```

Plus a platform-level boolean `users.is_super_admin`. Role assignment is per-org via `organization_members.role`.

### Super admin
**What they can do:** see and edit everything across all orgs; access `/admin/orgs`, `/admin/users`, `/admin/subscriptions`, `/admin/usage`; manually override plans, seat counts, role assignments; bypass cross-org guards.
**Toggle behavior:** super admins still log in like any other user; the role switcher in the header dropdown lets them act as a member of any org they're enrolled in. Inferred: they bypass plan gating (`session.user.isSuperAdmin` is checked in cross-org checks).

### Organization admin
**What they can do:**
- Manage org members + invites at `/org/members`
- Edit org settings + cert template at `/org/settings`, `/org/cert-template`
- View billing + Stripe Customer Portal at `/org/billing`
- See dashboard with token-usage + MTD cost panel
- See every program in the org (whether they own it or not — Inferred from RLS pattern)
**Pages they access:** the full `/org/*` subtree, the `/admin/*` subtree is super-admin-only.
**Subscription / seat restrictions:** pulls from the org's `plans` row — `instructor_seats`, `learner_seats`, `monthly_token_budget`, `features`.

### Instructor (Creator)
**What they can do:**
- Create / edit / duplicate / delete their own programs (Chatrails)
- Author bot nodes; configure system prompts, models, KB, rubrics
- Invite learners + co-instructors + TAs
- Grade transcripts
- Build cert templates + issue cert awards (gated by plan `certificates: true`)
- Use AI Chatrail Generator
**Pages:** `/programs`, `/programs/new`, `/programs/generate`, `/programs/[id]/*`, `/dashboard`, `/rubrics`, `/account`.
**Subscription restrictions:** plan-feature-gated via `lib/billing/gate.ts` — `canCreateProgram`, `canCreateChatbotNode`, `canIssueCertificate`, `canSeatLearner`, `canSeatInstructor`, `allowedModelsForPlan`. Free plan caps at 1 program / 3 chat nodes / 5 learners / no certs.

### Teaching Assistant (TA)
**What they can do:**
- Co-grade with the instructor
- Read and act on submissions for the program(s) they're attached to
- See the gradebook, the bot nodes, the roster
- Cannot create or delete programs (Inferred — author actions check `instructor | org_admin | super_admin` only)
**Pages:** Likely the same as instructor for the programs they're attached to via `program_instructors.capacity = 'ta'`.

### Learner / Student
**What they can do:**
- See programs they're enrolled in via `/learn`
- Walk a journey at `/learn/[programId]`
- Chat with bot nodes at `/learn/[programId]/[nodeId]`
- Submit, view grades, view rubric breakdown
- Earn certificates and download PDFs from `/learn/certificates`
**Cannot do:** see system prompts (RLS on `chatbot_configs` is staff-only; learners read a stripped-down `chatbot_learner_configs` view), see other learners' submissions, see draft/archived Chatrails.

### Public visitor
- Marketing pages: `/`, `/features`, `/pricing`, `/for-education`, `/for-corporate`, `/aup`, `/privacy`, `/terms`
- Cert verification: `/verify-cert/[code]`
- Auth: `/login`, `/signup`, `/forgot-password`, `/accept-invite/[token]`
- Cannot access any `/dashboard`, `/programs`, `/learn`, `/org`, `/admin` route — middleware bounces to `/login`

### Demo user
**Status:** seeded test users exist (super admin, org admin, instructor, TA, learner per `supabase/seed.sql`) but there's no separate "demo" role. The seed file is dev-only and not safe to run on production.

### Toggle / role-switching behavior
The header has a role switcher that swaps the "active" view among the orgs/roles a user belongs to via a cookie override (per `lib/auth/active-role.ts`). A user with multiple memberships sees both views and can context-switch without re-logging-in.

## 10. User Journey

### Instructor / Creator
1. **First visit** → marketing landing → `/pricing` → click "Free" or self-serve plan
2. **Signup** at `/signup` with intent = "Instructor / Trainer / SME" → auto-creates a personal org

8. **Grading:** `/programs/[id]/gradebook` → filter PENDING → click cell → grader panel slides in → AI suggest → adjust per-criterion → SAVE → learner notified
9. **Lifecycle:** PUBLISH / UNPUBLISH / ARCHIVE / RESTORE controls in overview → DELETE CHATRAIL with title-typed confirmation when abandoned
10. **Billing:** `/org/billing` → Stripe Checkout → upgrade if hitting caps; Customer Portal for plan / card management

### Learner / Student
1. **Receive invite** → email with link OR direct add (if instructor's account was already in the system)
2. **Accept invite** at `/accept-invite/[token]` → either signup (logged out) or claim-for-current-account (logged in, just shipped) → friendly mismatch path if email differs
3. **Land on `/learn`** → cassettes for each program with progress, due dates, last activity, cert count
4. **Open a Chatrail** at `/learn/[programId]` → progress bar + NEXT STEP banner + node grid with status badges (DONE / ACTIVE / AVAILABLE / LOCKED / RETRY) + ATTEMPTS · X/Y
5. **Enter a node** → for bots: chat screen with path rail (with LOCKED awareness), attempt counter, token meter, instructions strip, BRIEFING chip, SEND/STOP
6. **Submit** → confirmation → submitted banner → bounce back to journey
7. **View grade** → `/learn/[programId]/[nodeId]/grade` → rubric breakdown, instructor comment, AI summary if shared, retry / review / download transcript / verify certificate

9. **Renewal / cancellation:** for learner-paid programs, Stripe-managed; for invited learners, instructor controls

### Org admin
1. **Signup** as `org_admin` intent or **claim** an admin invite from someone else's org
2. **Dashboard** at `/dashboard` → token usage + MTD cost panel + Chatrails grid + members button
3. **`/org/members`** → manage roles, suspend / reactivate
4. **`/org/billing`** → Stripe Checkout / Customer Portal for plan changes
5. **`/org/cert-template`** → customize signer, body text, logo upload
6. **Active phase** → review usage week to week, escalate to higher plan if approaching cap
7. **Renewal:** Stripe handles subscription billing; webhook syncs `plan_code` and `subscriptions.status`; failed payment moves org to `past_due` and freezes new enrollments while preserving existing work

### Super admin
1. Signed up the platform first (manual SQL flag); subsequently lands on the super-admin dashboard
2. Browses `/admin/orgs`, `/admin/users`, `/admin/subscriptions`, `/admin/usage`
3. Manual seat-count / plan / flag overrides as needed
4. Reviews the audit ledger (UI planned, raw rows queryable via SQL)

### Public visitor / unauthenticated buyer
1. Lands on `/` → reads pillar features → clicks `/pricing` → reads the live plan rows
2. Clicks **GET STARTED** → `/signup` → onboarding
3. Or clicks `for-education` / `for-corporate` for segment-specific framing
4. Or visits `/verify-cert/[code]` to confirm a certificate someone showed them

### Authentication flow
1. User signs up via `/signup` → `supabase.auth.signUp()` returns user_id
2. A trigger in `00010000000002_identity.sql` inserts a row into `public.users` mirroring `auth.users`
3. Server actions in `app/(auth)/actions.ts` provision org membership based on intent (instructor/org_admin/learner/ta) or invite token
4. Middleware (`middleware.ts`) reads the Supabase cookie session on every gated request
5. `lib/auth/active-role.ts` resolves the active org + role from cookie + memberships
6. RBAC checks in server actions gate sensitive ops (`requireSessionUser`, `getActiveRole`, role allow-lists)
7. RLS policies in `00010000000009_policies.sql` (and `00010000000010_phase_a_security.sql`) enforce row-level access at the DB layer

### Tenant / organization structure
- The canonical tenant is `organizations`; everything is `organization_id`-scoped via FK
- Users belong to many orgs via `organization_members` with one role per (org, user)
- Programs belong to one org; nodes / configs / KB / conversations / submissions / grades cascade off the program
- Cross-org actions are blocked at the gate AND at RLS (defense in depth)

### How data moves
- **Authoring:** server action → service role insert into `programs` / `path_nodes` / `chatbot_configs` → revalidate
- **Chat:** client POST → SSE route → admin client (RLS-bypass) reads bot config + KB + history → calls LLM provider (server-only key) → streams SSE deltas → on completion persists assistant message + updates conversation token totals + writes `usage_logs` row
- **Grading:** client opens grader panel → client supabase reads transcript (RLS-allowed for staff) → client POSTs to `/api/grade/suggest` → server uses admin client to read rubric + criteria + transcript → calls `suggestRubricScore` LLM helper → returns JSON → instructor edits + saves grade via server action → action mirrors status to `conversations`, runs `maybeAwardCertificates`, fires a `grade_returned` notification
- **Cert award:** server-side check on every grade save + mark-complete → if criteria met, insert `certificate_awards` row → email learner + create notification → learner downloads PDF from `/api/certificates/[id]/pdf?v=[verification_code]`

## 13. Data Model

All tables are in the `public` schema; RLS is enabled on every tenant table.

### Identity (`00010000000002_identity.sql`)
- **`users`** — profile mirroring `auth.users`. `is_super_admin` boolean.
- **`organizations`** — tenant root. `slug`, `org_type`, `plan_code`, `stripe_customer_id`.
- **`organization_members`** — (org, user, role) — many-to-many with role per relationship.
- **`invites`** — email + token + role + program_id (optional). Status: pending / accepted / revoked / expired.
- **`csv_imports`** — record of bulk CSV imports.
- **`audit_logs`** — actor / action / target / metadata; written by sensitive server actions.

### Curriculum (`00010000000003_curriculum.sql`)
- **`programs`** — top-level Chatrail. `title`, `description`, `status` (draft/published/archived), `default_model`, `passing_threshold`, `monthly_token_budget`, `learner_pays`, `learner_price_cents`, `share_conversations_with_org_admin`.
- **`program_instructors`** — (program, user, capacity) where capacity ∈ owner / co_instructor / ta.
- **`program_enrollments`** — (program, user) with `status` (active / suspended).
- **`path_nodes`** — every node. `type` (bot / content / pdf / slides / link / milestone / cert), `title`, `display_order`, `points`, `available_at`, `due_at`, `is_required`, `config` jsonb (type-specific).
- **`path_edges`** — (source, target, condition jsonb) with kinds: `null` (default — source must be completed), `after`, `min_score`, `date`, `either`.
- **`node_rules`** — extra release rules per node (open_on_date, after_prereq, min_score, either_or, branching, skip).

### AI (`00010000000004_ai.sql`)
- **`chatbot_configs`** — one per bot node. `system_prompt`, `learner_instructions`, `model`, `temperature`, `max_tokens`, `token_budget`, `attempts_allowed`, `rubric_id`, `ai_grading_enabled`, `use_program_kb`, `node_kb_id`. Plus `bot_name`, `avatar_initials`, `conversation_goal`, `completion_criteria`.
- **`knowledge_collections`** — program-scoped (one default per program created automatically).
- **`knowledge_files`** — uploaded file refs + `storage_path` + `status` (uploading / parsing / chunking / embedding / indexed / failed).
- **`document_chunks`** — text + page + offset metadata.
- **`embeddings`** — `vector(1536)` pgvector column.

A learner-safe **view** `chatbot_learner_configs` excludes `system_prompt`, `conversation_goal`, `completion_criteria` (added in Phase A security migration).

### Conversations (`00010000000005_conversations.sql`)
- **`conversations`** — one per (learner, node, attempt). `status` (not_started / in_progress / submitted / graded / completed / needs_revision), `total_prompt_tokens`, `total_completion_tokens`, `ai_summary`, `submitted_at`.
- **`conversation_messages`** — role + content + per-message token counts + citations jsonb + model.
- **`submissions`** — snapshot at submit time. `delivery_status` (on_time / late), FK to conversation + program + node + learner + org.
- **`analytics_events`** — generic event log (Inferred — schema-only; usage may be partial).

### Grading (`00010000000006_grading.sql`)
- **`rubrics`** — name + description + total_points + is_visible_to_learners.
- **`rubric_criteria`** — name + description + max_points + display_order.
- **`rubric_levels`** — label + points + description + display_order (per criterion).
- **`grades`** — one per submission. `status` (pending_review / graded / needs_revision / excused / not_submitted), `score`, `max_score`, `percentage`, `instructor_comment`, `ai_summary`, `ai_suggested_score`, `rubric_id`, `graded_at`, `graded_by`.
- **`rubric_scores`** — per-criterion: grade_id + criterion_id + score + comment.

### Certificates (`00010000000007_certificates.sql`)
- **`certificate_templates`** — per org. Signer name, body text, logo URL, paper size.
- **`certificates`** — definition attached to a node (which milestone triggers issuance, with min grade %).
- **`certificate_awards`** — one per (learner, program, certificate). `verification_code` (unique), `awarded_at`. PDF generated on demand from `/api/certificates/[id]/pdf`.

### Billing (`00010000000008_billing.sql`)
- **`plans`** — code, name, scope (instructor / organization), monthly_price_cents, instructor_seats, learner_seats, monthly_token_budget, features jsonb.
- **`subscriptions`** — Stripe customer + sub IDs, status (trialing / active / past_due / canceled / unpaid / incomplete), current_period_end.
- **`seat_pools`** — org-owned or instructor-owned seat banks (Inferred — schema only, full UX is roadmap).
- **`seat_assignments`** — link seat to user + program (Inferred).
- **`billing_events`** — Stripe webhook event log.
- **`usage_logs`** — one row per LLM call. organization_id, program_id, node_id, conversation_id, user_id, kind (chat / chat-preview / embedding / grade_suggest), model, prompt_tokens, completion_tokens, est_cost_usd.

### Notifications (`00010000000040_phase_t_notifications.sql`)
- **`notifications`** — user-scoped. Kind (invite_received / grade_returned / cert_awarded / etc.), title, body, href, metadata jsonb, read_at.

### Other
- **Indexes:** Phase H added 5 targeted indexes for gradebook + analytics joins (`00010000000020_phase_h_perf_indexes.sql`).
- **Default cert templates:** seeded in `00010000000030_phase_s_default_cert_templates.sql`.

## 14. AI Usage

### Where AI is used
1. **Chat streaming** (`/api/chat/stream`) — the primary product surface; learner ↔ bot conversations
2. **Preview chat** (`/api/chat/preview`) — staff-only, ephemeral, no DB writes
3. **AI grading suggestion** (`/api/grade/suggest`) — summarizes transcript + suggests per-criterion rubric scores; instructor reviews + overrides
4. **AI Chatrail generator** (`/programs/generate` + server action) — creator brief → multi-node Chatrail with prewritten system prompts and learner instructions
5. **Embeddings** — KB documents are chunked, embedded (OpenAI 1536 or Gemini 768→1536 padded), stored in pgvector, retrieved on every chat turn (when `use_program_kb=true`)

### Model providers used / planned
- **Anthropic Claude** — `claude-haiku-4-5` (default), `claude-sonnet-4-6`, `claude-opus-4-7`, `claude-3-5-sonnet-latest`, `claude-3-5-haiku-latest`
- **OpenAI** — `gpt-4o`, `gpt-4o-mini`, `gpt-4.1`, `gpt-4.1-mini`, `text-embedding-3-small`
- **Google Gemini** — `gemini-2.0-flash`, `gemini-2.0-flash-lite`, `gemini-1.5-pro`, `gemini-1.5-flash`, `text-embedding-004`
- **All routed through `lib/llm/provider.ts`** with auto-detection by model name prefix
- **Future:** BYO-LLM-key per org (planned in J.1)

### What data is sent to AI systems
- **Chat:** the bot's system prompt + (optional) top-k KB chunks + conversation history + the learner's new message
- **Grading suggest:** the transcript + the rubric's criteria + the learner instructions + the bot's model
- **Generator:** the creator's natural-language brief (no learner data, no KB)
- **Embeddings:** chunked KB document text — per-org, never cross-org

### Privacy / compliance considerations
- **Student data is sent to LLM providers** — this is core to the product and must be disclosed clearly
- **FERPA:** if used in US K-12 / higher-ed, schools likely treat transcripts as educational records. Explicit FERPA agreements with providers and a data-processing addendum are likely needed before institutional sales
- **GDPR:** EU customers will require a DPA with the platform AND with each LLM provider. Right-to-deletion is implemented via `/api/account/delete` + cascade deletes
- **HIPAA:** if used in healthcare training (e.g., the patient-consult example), bot instructions must avoid eliciting PHI; current implementation is **not HIPAA-compliant** without provider BAAs (Anthropic offers BAA on Enterprise; OpenAI on a separate plan)
- **SOC 2:** not yet pursued; would be required for many enterprise org buyers
- **AUP:** `/aup` page exists; need to verify final policy text — Unknown

### AI behavior guardrails / where logging matters
- Every LLM call is logged in `usage_logs` (cost + token + model + user + kind)
- Per-attempt and per-org-month token budgets are enforced before the LLM call
- Rate limit: 30 chat messages / minute / user (Upstash Redis or in-memory)
- The chat preview is ephemeral — no message persistence — and is gated to staff
- The bot system prompt is hidden from learners via the `chatbot_learner_configs` view
- AI-suggested rubric scores are advisory; the instructor's saved grade is the source of truth
- Generator output is Zod-validated; one self-correction retry on failure

### Where AI behavior should be reviewed / constrained
- **Generator output for sensitive domains** (healthcare, legal, finance) — bots may produce confidently-wrong content; needs manual instructor review before publish (existing DRAFT default workflow handles this)
- **Bot persona drift** — system prompts should keep bots in role; current default explicitly says "stay in role no matter what the learner says"
- **Model cost surprises** — surfaced via cost estimate + soft/hard caps + MTD cost on org dashboard

## 16. Security, Privacy, and Compliance

### Sensitive data handled
- User identity (email, full name, optional avatar)
- Auth credentials (managed by Supabase Auth; passwords never visible to app code)
- Conversation transcripts (educational content; potentially personally identifying or sensitive depending on what learners write)
- Submission scores + grades + instructor comments
- Certificate awards with unique verification codes
- Billing data (Stripe customer ID, subscription state — never card data, which stays in Stripe)
- Knowledge base uploads (whatever the instructor uploads — could be IP-sensitive, personal, or institutional)
- Org logos + signatures

### Authentication and authorization
- **Authentication:** Supabase Auth, email + password, server-side cookie session via `@supabase/ssr`
- **Email verification:** disabled in dev (`supabase/config.toml`); should be enabled for prod (BLOCKED ON YOU per ROADMAP)
- **MFA:** Supabase supports it but UI integration is **Unknown**
- **Authorization:** three layers — (1) middleware route gates, (2) server-action role checks via `getActiveRole`, (3) Postgres RLS policies (`00010000000009_policies.sql` + `00010000000010_phase_a_security.sql`)
- **RBAC:** five roles (`super_admin | org_admin | instructor | ta | learner`) with org-scoped membership

### Role-based access control
- Cross-org access blocked at gate AND RLS (defense in depth)
- Learners cannot read bot system prompts (separate `chatbot_learner_configs` view)
- Staff (instructor / TA / org_admin in the org, or super_admin) bypass learner-only guards
- Permanent delete requires title-typed confirmation
- Sensitive server actions write `audit_logs` rows

### Payment data handling
- Card data never touches Chatrail servers — handled entirely by Stripe Checkout / Customer Portal
- Stripe customer + subscription IDs stored on `organizations` and `subscriptions`
- Webhook signature verification implemented in `/api/stripe/webhook`

### Compliance posture
- **FERPA** (US educational records): not yet formally compliant; institutional sales need DPAs + provider BAAs. Inferred — no FERPA addendum present.
- **GDPR** (EU): right-to-deletion via `/api/account/delete` + cascade FKs; right-to-export via `/api/account/export`. Full DPA wording in `/privacy` should be reviewed by counsel before EU sales.
- **HIPAA** (health data): **not compliant** without provider BAAs; should not be used for PHI today.
- **SOC 2:** not pursued.
- **COPPA** (under-13 US): not addressed; Chatrail should not be used by under-13s without parental consent + institutional process.

### Security gaps / unknowns
- **Email confirmation off in dev** — BLOCKED ON YOU to enable for prod
- **Custom domain + DNS / SPF / DKIM / DMARC** — not yet set up
- **Bug bounty / vuln-disclosure** — only `security@chatquest.app` in `SECURITY.md`; mailbox status Unknown
- **Service-role key in `.env.local`** — common dev pattern but should be rotated on deployment per ROADMAP
- **Audit log viewer UI** — rows are written but the UI is Unknown / planned
- **MFA enforcement** — Supabase supports it; not enforced

### Recommended improvements (security)
1. Enable Supabase email confirmation for production
2. Set up DKIM + SPF + DMARC on the sending domain
3. Get a custom domain + valid TLS chain
4. Rotate the seed-test service role key
5. Build the `/admin/audit` viewer

## 18. Known Issues and Risks

### Technical risks
- **LLM provider concentration** — Anthropic outage, Gemini outage, or OpenAI outage each take down a model class; the multi-provider adapter mitigates this but per-bot model choice is still single-vendor
- **Postgres single-region** — Supabase single-region; latency for distant regions; backup / DR Unknown
- **Netlify Lambda limits** — 26s default timeout could clip slow LLM calls (the generator can take 15–40s); need to verify Netlify function timeout config
- **Direct Postgres flakiness on Lambda** — already burned us once on `/pricing`; pgvector RPC fix was Phase A; future Drizzle migrations should avoid relying on outbound :5432 from Lambdas
- **No background job queue** — KB indexing, email digests, big exports run inline today

### Business risks
- **AI tutoring is a hot, crowded category** — fast-moving competition from Anthropic / OpenAI / Google directly, plus countless YC startups. Differentiation needs to be lived, not just claimed.
- **Buyer persona is split** between solo creators (low ACV, high churn) and institutions (high ACV, slow sales cycles). Pricing reflects both but the GTM motion is two motions, not one.
- **K-12 sales** require trust signals (FERPA, COPPA, accessibility audits, district procurement) Chatrail does not yet have.
- **Higher-ed sales** require LTI integration with Canvas / Blackboard / Moodle (not built).
- **Token cost in COGS** — at high volume, LLM cost is most of variable cost; misuse / cheat-prompts could shred margin without strict caps.

### UX risks
- Path builder on mobile is acknowledged-weak
- The AI generator's 15–40s wait is brittle perception-wise even with progress messages — streaming the output would feel snappier (not yet done)
- Brutalist design is polarizing; might need a softer variant for institutional buyers (Inferred — no theming exists)

### Security risks
- Email confirmation off in dev (must enable for prod)
- Service-role key has been pasted to chat per ROADMAP — must be rotated
- CSP is report-only — needs to flip to enforcing
- No MFA enforcement for admins
- Audit log viewer UI not built — investigations require SQL

### Legal / compliance risks
- FERPA / GDPR / HIPAA posture is informal; institutional sales blocked until DPAs + BAAs in place
- AUP / Terms / Privacy text is scaffold — needs counsel review
- AI training opt-out / data-residency claims with each LLM provider need formal documentation before enterprise sales
- DMCA / takedown process not formalized
- Accessibility audit (WCAG 2.1 AA) not yet done — many institutions require it

### Market risks
- Anthropic / OpenAI could ship competing first-party LMS surfaces
- Khan Academy / Coursera could launch chatbot-native modes
- Existing LMS incumbents (Canvas / Blackboard) could close the gap with bolt-ons that are "good enough" for the median buyer

### Cost risks
- LLM token cost per learner per Chatrail is the biggest variable cost; if the generator + preview + grading + chat all use Sonnet, a single learner attempt can hit $0.50+. At 10K-learner cohorts, this is real money.
- Free tier (50K tokens) abusers could chain accounts to grind on the free product — partially mitigated by rate limits, but not abuse detection

### AI usage risks
- Bots producing confidently-wrong content in sensitive domains (legal, medical, financial) without instructor review
- Prompt injection via learner messages → bot reveals system prompt / changes role → mitigated by "stay in role" defaults but not adversarially tested
- Learners using bots to do their thinking for them (the meta-risk for AI tutoring as a category) — partially mitigated by transcript grading, but a real conversation about academic integrity belongs in the AUP

### Code quality issues visible from the repo
- Some inline relative-time copies were extracted in this session, but the codebase still has duplication across cassette renderers (instructor dashboard / programs list / /learn) that could share more
- A few `unknown as` casts in places where the Supabase TS types don't match the joined select shape — fine but brittle if schema changes
- The legacy package name `chatquest` vs product name `Chatrail` is a minor branding inconsistency
- Some TypeScript linter warnings about exhaustive deps were silenced inline — Inferred
